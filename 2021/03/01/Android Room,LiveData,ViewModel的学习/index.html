Android Room,LiveData,ViewModel的学习

在第一行代码的学习中，android提供了与内嵌式数据库sqlite的API，也提供了同样为持久化技术的SharedPrefences。
两者各有优劣，SharedPrefences交互方便但是使用xml的格式存储，可以被外部查看，数据存储不安全。
sqlite的缺点在于交互不便，于是广大的开发者大军写出了很多关于sqlite的开源库，也包括第一行代码作者guolin大神在书中提及自己写出的litepal。同时google也推出了自己的数据库映射库（暂且这么称吧）。
通过Android Training的Lesson10关于Room,LiveData,ViewModel的使用,我对Room等有了初步的理解。
下文将会讲述一下比较方便的Room使用方法

这张图很清晰地说明了Room的实现以及便捷调用

  1. Entity 实现数据库一张表的实际类
  2. Dao 编写调用数据库的接口
  3. RoomDatabase 实现这个数据库的实际类
  4. Repository 整理Dao接口的API
  5. LiveData 与 ViewModel则与和具体的逻辑有关

Entity 实现数据库一张表的实际类

Entity代表了这是一个实体类，tableName是表名，并用mWord来作为word_table表的主键（@PrimaryKey），name设置列名，typeAffinity设置这一列的类型
在我的观察下，Room使用了大量的注解，关于Java语言注解（annotation）是我Java知识的漏洞，可以作为之后一段时间的学习内容。

Dao 编写调用数据库的接口

Dao代表了这是一个可操作的接口类
每个方法上边标注的注解与SQL语句代表这些方法的操作
熟悉MySql的同学一定很容易就能明白这些，其中getAnyWord这个方法只会返回最多一个对象，是用来判断数据库是否为空的方法

RoomDatabase 实现这个数据库的实际类

RoomDatabase编写就复杂很多了，首先要实现一个继承于RoomDatabase的抽象类。
@Database(entities = {Word.class}, version = 1, exportSchema = false)代表了这是一个数据库类，entities的花括号要包含这个数据库中包含的表
同时基于设计模式单例模式与逻辑考虑，建议只有一个数据库对象即上面代码所实现的那个方法，同时注意到这个方法是synchronized（同步的）因为对于数据的库的访问不能在主线程中进行，这也是设置同步方法和后面Repository需要实现的原因
最最重要的是在这个类中要提供接口的方法也就是wordDao();
至于后面的onOpen并不是必须要实现的回调

Repository 整理Dao接口的API

关于上面的内容其实有人要问了，Dao接口已经实现很方便的接口，为什么要加一个Repository来整理接口了，不是多此一举吗
原因就在于访问数据库不能放在主线程，所以Dao的接口还不能直接调用，得放在自己编写的Thread类或者AsyncTask类里面，所以需要一个Repository来整理Dao接口的API

如上所示，Repository把其他线程的操作进行做了整理这样写代码的人才能更方便地访问数据库

关于LiveData 与 ViewModel 在本文就不多阐述，它是用来与UI界面交互的类
